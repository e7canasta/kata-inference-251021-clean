TheÂ `InferenceHandlerFactory`Â is responsible for creating inference handlers based on the ROI mode configuration. It encapsulates the decision logic for selecting and constructing the appropriate handler type (`StandardInferenceHandler`,Â `AdaptiveInferenceHandler`, orÂ `FixedROIInferenceHandler`) along with their associated dependencies (ROI state, model, inference configuration).

For information about how these handlers process video frames, seeÂ [Inference Pipeline](https://deepwiki.com/acare7/kata-inference-251021-clean4/5-inference-pipeline). For details on ROI strategies themselves, seeÂ [ROI Strategies](https://deepwiki.com/acare7/kata-inference-251021-clean4/5.1-roi-strategies). For information about the broader factory pattern system, seeÂ [Factory Pattern System](https://deepwiki.com/acare7/kata-inference-251021-clean4/3.3-factory-pattern-system).

## Purpose and Responsibilities

TheÂ `InferenceHandlerFactory`Â centralizes handler construction logic that was previously embedded in theÂ `InferencePipelineController`. Its primary responsibilities are:

|Responsibility|Description|
|---|---|
|**ROI Mode Validation**|Determines which handler type to create based onÂ `ROI_MODE`Â configuration (`none`,Â `adaptive`,Â `fixed`)|
|**ROI State Creation**|Delegates toÂ `validate_and_create_roi_strategy()`Â to construct appropriate ROI state objects|
|**Model Creation**|Creates YOLO model instances (local ONNX or Roboflow) usingÂ `get_model_from_config()`|
|**Handler Assembly**|Constructs the appropriate handler with all required dependencies|
|**Configuration Mapping**|Maps configuration parameters to handler-specific settings|

The factory implements a single static methodÂ `create()`Â that returns a tupleÂ `(handler, roi_state)`.

## Handler Types Overview

The factory can create three distinct handler types, each with different capabilities:

|Handler Type|Class|ROI Mode|Toggle Support|Description|
|---|---|---|---|---|
|**Standard**|`StandardInferenceHandler`|`none`|No|No ROI processing; pipeline usesÂ `model_id`Â directly withÂ `.init()`|
|**Adaptive**|`AdaptiveInferenceHandler`|`adaptive`|Yes|Dynamic ROI adjustment based on detections; supports runtime enable/disable|
|**Fixed**|`FixedROIInferenceHandler`|`fixed`|No|Static ROI with pre-configured coordinates; immutable|

All handlers implement theÂ `BaseInferenceHandler`Â abstract interface, ensuring consistent behavior across the system.

## ROI Mode Decision Logic

The factory's primary decision point is theÂ `ROI_MODE`Â configuration value, which determines the entire construction path:

### Mode:Â `none`

WhenÂ `config.ROI_MODE == 'none'`:

- CreatesÂ `StandardInferenceHandler`Â [adeline/inference/handlers/standard.py22-41](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/handlers/standard.py#L22-L41)
- ReturnsÂ `roi_state = None`
- Pipeline uses standard initialization:Â `InferencePipeline.init(model_id=...)`
- No custom logic callback is provided to the pipeline

### Mode:Â `adaptive`Â orÂ `fixed`

WhenÂ `config.ROI_MODE`Â isÂ `'adaptive'`Â orÂ `'fixed'`, the factory enters the "custom logic path":

1. **ROI Configuration Construction**Â [adeline/inference/factories/handler_factory.py82-100](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L82-L100)
    
    - CreatesÂ `ROIStrategyConfig`Â with all adaptive and fixed parameters
    - IncludesÂ `imgsz`Â for model size coordination
2. **ROI State Validation and Creation**Â [adeline/inference/factories/handler_factory.py102-105](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L102-L105)
    
    - CallsÂ `validate_and_create_roi_strategy(mode, config)`
    - Validates parameters (e.g.,Â `x_min < x_max`, normalized coordinates)
    - ReturnsÂ `ROIState`Â for adaptive orÂ `FixedROIState`Â for fixed
3. **Model Creation**Â [adeline/inference/factories/handler_factory.py107-114](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L107-L114)
    
    - UsesÂ `get_model_from_config()`Â to create YOLO model
    - Supports both local ONNX models and Roboflow models
    - AppliesÂ `imgsz`Â parameter for model input size
4. **Inference Configuration**Â [adeline/inference/factories/handler_factory.py116-120](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L116-L120)
    
    - CreatesÂ `ModelConfig`Â withÂ `confidence`Â andÂ `iou_threshold`
5. **Process Function Selection**Â [adeline/inference/factories/handler_factory.py122-123](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L122-L123)
    
    - Determines the appropriate processing function based on model type
6. **Handler Construction**Â [adeline/inference/factories/handler_factory.py125-150](https://github.com/acare7/kata-inference-251021-clean4/blob/a0662727/adeline/inference/factories/handler_factory.py#L125-L150)
    
    - CreatesÂ `AdaptiveInferenceHandler`Â orÂ `FixedROIInferenceHandler`
    - Passes all dependencies to handler constructor

TheÂ `InferenceHandlerFactory`Â is called by theÂ `PipelineBuilder`Â during the first phase of pipeline construction:


```mermaid

## Validation and Error Handling

The factory implements fail-fast validation at multiple levels:

### ROI Mode Validation

```

### Delegated Validation

The factory delegates most validation to specialized functions:

|Validation Type|Delegated To|Validated Items|

|**ROI Parameters**|`validate_and_create_roi_strategy()`|Normalized coordinates, parameter consistency|

|**Fixed Coordinates**|`FixedROIState.__init__()`|`x_min < x_max`,Â `y_min < y_max`, rangeÂ `[0.0, 1.0]`|

## Design Rationale

TheÂ `InferenceHandlerFactory`Â embodies several key design principles:

### Separation of Concerns

|Concern|Owner|
|---|---|
|**When to construct**|`PipelineBuilder`|
|**How to construct**|`InferenceHandlerFactory`|
|**What to construct with**|`validate_and_create_roi_strategy`,Â `get_model_from_config`|
|**How to use**|`InferencePipelineController`|

The factory knowsÂ **how**Â but notÂ **when**Â orÂ **why**. The builder knowsÂ **when**Â but notÂ **how**.

### Centralized Decision Logic

### Fail-Fast Validation

All validation happens at construction time, not runtime. Invalid configurations are rejected before the pipeline starts processing video.

### Type Safety

The factory returns typed objects (`BaseInferenceHandler`, not genericÂ `Any`), enabling static type checking and IDE autocomplete.
